# /리뷰5 — 5단계 코드 리뷰

커밋 또는 PR 전에 5단계로 코드를 검증한다.
모든 단계를 통과해야 "리뷰 완료"로 판정.

---

## 1단계: 타입 안정성 검사

```bash
npx tsc --noEmit
```

- `any` 타입 사용 여부 확인
- `as` 타입 단언 사용 여부 확인 (`as const` 제외)
- `@ts-ignore`, `@ts-expect-error` 사용 여부 확인
- DB 타입이 Supabase 생성 타입(`Database`)에서 파생되었는지 확인

**판정:** 위반 0건이면 ✅, 있으면 ❌ + 위반 목록 제시

---

## 2단계: 보안 검사

- `.env*` 파일이 staged 되어 있지 않은지 확인
- 코드 내 하드코딩된 키/토큰 검색 (`eyJ`, API key 패턴)
- `service_role` 키가 클라이언트 코드에 노출되지 않는지 확인
- RLS 정책이 비활성화된 테이블이 없는지 확인 (마이그레이션 파일 검사)
- Webhook 엔드포인트에 서명 검증이 있는지 확인

```bash
bash scripts/check-infra.sh
```

**판정:** 보안 위반 0건이면 ✅, 있으면 ❌ + 즉시 수정 필요

---

## 3단계: 코드 품질 검사

```bash
npx eslint . --max-warnings=0
npx prettier --check .
```

- ESLint 경고 0건 확인
- Prettier 포맷 일치 확인
- 한 파일 500줄 초과 여부 확인
- 불필요한 `console.log` 검색
- 사용하지 않는 import/변수 확인

**판정:** 위반 0건이면 ✅, 있으면 자동 수정 후 재확인

---

## 4단계: 아키텍처 검사

변경된 파일들을 대상으로 다음을 검사:

- 폴더 구조가 `docs/conventions.md` 규약을 따르는지
- 컴포넌트 파일명이 PascalCase인지
- 유틸 파일명이 kebab-case인지
- 서버/클라이언트 경계가 올바른지 (`'use client'`, `'use server'`)
- DB 쿼리에 `tenant_id` 필터가 빠지지 않았는지 (멀티 테넌트 격리)
- RLS에만 의존하지 않고 애플리케이션 레벨에서도 tenant 체크

**판정:** 구조 위반 0건이면 ✅, 있으면 ❌ + 수정 방안 제시

---

## 5단계: 학습 교훈 대조

`docs/LESSONS.md`를 읽고, 이번 변경이 기록된 교훈을 위반하지 않는지 확인.

```bash
bash scripts/check-lessons.sh
```

- 3회 이상 재발한 교훈(자동 검증 룰 활성)과 대조
- 신규 패턴의 실수가 있다면 `/교훈추가` 권고

**판정:** 교훈 위반 0건이면 ✅, 있으면 ❌ + 해당 교훈 ID 제시
