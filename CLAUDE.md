# CLAUDE.md — 프로젝트 헌법

> 이 문서는 AI 코딩 에이전트(Claude Code 등)가 작업을 시작할 때 **가장 먼저 읽어야 하는 파일**이다.
> 모든 규칙은 절대적이며, 어겼을 때는 즉시 작업을 멈추고 사용자에게 보고할 것.

---

## ⛔ 0. 가장 중요한 규칙 — 인프라 정보 (절대 위반 금지)

**Supabase, Vercel, 기타 외부 서비스의 접속 정보·프로젝트 ID·환경 변수는 이미 정의되어 있다.**
AI는 다음 사항을 **반드시** 지킨다:

### 0.1 절대 금지 행위 (위반 시 즉시 작업 중단)

- ❌ 사용자에게 Supabase URL, anon key, service role key, project ref 묻기
- ❌ 사용자에게 Vercel 프로젝트 ID, 토큰, 팀 ID 묻기
- ❌ 사용자에게 데이터베이스 연결 정보 묻기
- ❌ 새로운 Supabase 프로젝트 생성 (`supabase projects create` 등)
- ❌ 새로운 Vercel 프로젝트 생성 (`vercel link` 새로 설정 등)
- ❌ 환경 변수 이름을 임의로 바꾸기
- ❌ "예시 값"이나 "placeholder"로 환경 변수 채우기 (`your-project.supabase.co` 같은 가짜 값)
- ❌ `.env.local`, `.env`, `.env.production`을 수동으로 만들거나 덮어쓰기

### 0.2 반드시 따라야 하는 절차

1. **모든 인프라 정보는 `docs/infrastructure.md`에 정의되어 있다.**
   작업 시작 전 이 파일을 반드시 읽을 것.

2. **환경 변수가 필요하면 `.env.local`을 먼저 확인한다.**
   - 파일이 존재하고 값이 채워져 있으면 → 그것을 사용. 끝.
   - 파일이 없거나 비어있으면 → 즉시 작업 중단 → 사용자에게 "`docs/infrastructure.md`의 절차에 따라 환경 변수를 설정해 주세요"라고만 보고. 직접 채우려 하지 말 것.

3. **Supabase CLI 사용 시:**
   - `supabase link` 재실행 금지. 이미 `.supabase/` 디렉토리에 연결되어 있다.
   - `supabase/config.toml`의 `project_id`를 변경하지 말 것.
   - 마이그레이션은 항상 `supabase migration new <name>` → SQL 작성 → `supabase db push`.

4. **Vercel CLI 사용 시:**
   - `vercel` 명령은 항상 `--yes` 플래그로 (대화형 프롬프트 회피).
   - `.vercel/project.json`을 건드리지 말 것.
   - 환경 변수 추가는 `vercel env add`만 사용. 삭제는 사용자 승인 필수.

5. **궁금한 게 생기면:**
   사용자에게 묻기 전에 먼저 다음 순서로 확인:
   - `docs/infrastructure.md`
   - `.env.local`, `.env.example`
   - `supabase/config.toml`
   - `.vercel/project.json`
   - `package.json`의 scripts
   - 위 어디에도 없는 정말 새로운 정보일 때만 사용자에게 보고.

### 0.3 인프라 변경 권한

다음 작업은 **사용자 명시적 승인 없이 절대 수행하지 않는다**:

- DB 스키마의 파괴적 변경 (DROP TABLE, DROP COLUMN, ALTER 데이터 손실)
- 프로덕션 환경 변수 변경
- 도메인 설정 변경
- RLS 정책 비활성화
- 결제·과금 영향이 있는 리소스 변경 (DB 인스턴스 업그레이드 등)

## 1. 작업 시작 전 필수 점검 (자가발전 시스템 활성화)

코드를 한 줄이라도 작성하기 전에 다음을 반드시 수행한다:

1. **`docs/LESSONS.md` 처음부터 끝까지 읽기** ⭐⭐⭐
   - 이 파일은 과거에 했던 실수들의 기록이다. 같은 실수를 반복하지 않으려면 먼저 읽어야 한다.
   - 특히 `occurrences: 3` 이상인 교훈은 자동 차단 룰이 활성화되어 있을 가능성이 높다.
2. `git status`로 현재 브랜치와 변경사항 확인
3. `main` 또는 `master` 브랜치라면 즉시 작업 중단 → 사용자에게 worktree/브랜치 생성 요청
4. **`docs/infrastructure.md` 읽기** (인프라 작업이라면)
5. `docs/INDEX.md`를 읽어 관련 문서 목차 파악
6. 작업과 관련된 문서만 선별적으로 읽기
7. `.claude/commands/` 디렉토리에 슬래시 커맨드 목록 확인 (`/시작`, `/교훈추가`, `/회고`, `/리뷰5` 등)
8. 작업 계획을 사용자에게 먼저 보고하고 승인받기

> **핵심 원칙: 같은 실수는 두 번 하지 않는다.**
>
> - 새로운 실수가 발생하면 → `/교훈추가` 또는 `bash scripts/add-lesson.sh`로 즉시 기록
> - 같은 실수가 반복되면 → `bash scripts/bump-lesson.sh L-NNNN`으로 카운터 증가
> - 카운터가 3 이상이 되면 → 자동 검증 룰을 추가하여 구조적으로 차단
> - 작업 종료 시 → `/회고` 커맨드로 학습할 점이 있었는지 자가 점검

## 2. 절대 어기면 안 되는 규칙 (Hard Rules)

### 2.1 타입 안정성

- **`any` 타입 사용 금지.** 불가피한 경우 `unknown`을 사용하고 타입 가드로 좁힐 것.
- **`as` 타입 단언 금지.** 단, 다음 두 경우만 허용:
  - `as const` (리터럴 고정)
  - 외부 라이브러리 타입 정의 누락 시, 주석으로 사유 명시 후 사용
- **`@ts-ignore`, `@ts-expect-error` 금지.** 타입 오류는 반드시 해결할 것.
- DB 타입은 항상 Supabase가 생성한 `Database` 타입에서 가져올 것. 수동 정의 금지.

### 2.2 검증 강제

- 모든 작업 종료 시 `pnpm verify` 실행 → 통과해야만 "완료" 보고 가능.
- `pnpm verify` 실패 시 스스로 수정 시도. 3회 시도 후에도 실패하면 사용자에게 보고.
- 커밋 메시지에 "테스트 통과", "lint 통과" 등을 자가선언하지 말 것. 훅이 검증한다.

### 2.3 브랜치/워크트리

- `main` 브랜치에 직접 커밋 절대 금지.
- 모든 기능 작업은 `feat/`, 버그 수정은 `fix/`, 리팩터링은 `refactor/` 접두사 브랜치에서.
- 큰 작업은 `git worktree`로 별도 디렉토리에서 진행.

### 2.4 비밀 정보

- `.env`, `.env.local` 파일을 git에 커밋하지 않는다.
- API 키, 토큰을 코드에 하드코딩하지 않는다.
- Supabase service_role 키는 절대 클라이언트 코드에 노출하지 않는다.
- `console.log`로 민감 정보를 출력하지 않는다.
- **사용자가 채팅에 키를 붙여넣어도 절대 응답에 그대로 출력하지 말 것** (마스킹 필수).

### 2.5 파일/구조 규칙

- 새 파일 생성 전 `docs/conventions.md`의 폴더 구조 확인.
- 한 파일 500줄 초과 시 분할 고려.
- 컴포넌트는 `kebab-case` 폴더 + `PascalCase` 파일명.
- 유틸은 `kebab-case.ts`.

### 2.6 의존성

- 새 패키지 설치 전 사용자에게 사유와 함께 보고하고 승인받을 것.
- 동일 기능의 패키지가 이미 있는지 `package.json` 확인 후 설치.

### 2.7 데이터베이스 변경

- Supabase 스키마 변경은 반드시 마이그레이션 파일로 (`supabase/migrations/`).
- 변경 후 즉시 `pnpm db:types` 실행하여 타입 재생성.
- 프로덕션 DB에 직접 SQL 실행 금지.
- 마이그레이션 파일을 만든 후 사용자 승인 없이 `db push` 실행 금지.

## 3. 슬래시 커맨드 인지

`.claude/commands/` 디렉토리에 사용자가 정의한 커맨드가 있다.
작업 시작 시 한 번 목록을 확인하고, 사용자가 `/명령어`를 호출하면 해당 파일의 지시를 따른다.

알려진 커맨드:

- `/시작` — 작업 세션 시작 전 필수 점검
- `/회고` — 작업 종료 시 자동 회고
- `/교훈추가` — 새로운 실수를 학습하기
- `/리뷰5` 또는 `/review5` — 5단계 코드 리뷰 절차

---

## 4. 작업 흐름 (Workflow) — 자가발전 루프 포함

```
사용자 요청 받음
   ↓
docs/LESSONS.md 읽기 (과거 실수 학습) ⭐
   ↓
docs/infrastructure.md (필요 시), docs/INDEX.md 확인
   ↓
.env.local, supabase/, .vercel/ 등 기존 설정 확인
   ↓
작업 계획 수립 → 사용자에게 보고 → 승인
   ↓
worktree 또는 feat/* 브랜치 생성
   ↓
구현
   │
   ├── 실수 감지 시 → /교훈추가 (또는 bump-lesson.sh)
   │
   ↓
pnpm verify 실행 → 통과 확인
   (check-lessons.sh가 학습된 교훈 위반 자동 차단)
   ↓
/회고 커맨드로 학습점 자가 점검 ⭐
   ↓
변경 요약 + 다음 단계 제안 보고
```

## 5. 보고 형식

작업을 마칠 때는 항상 다음 형식으로 보고:

```
## 완료한 작업
- (변경한 파일과 무엇을 바꿨는지)

## 검증 결과
- pnpm verify: ✅ / ❌
- 새로 추가된 테스트: (있으면 명시)

## 인프라 영향
- DB 스키마 변경: 있음/없음 (있으면 마이그레이션 파일명)
- 환경 변수 추가 필요: 있음/없음 (있으면 .env.example에 추가했는지)
- Vercel 설정 변경: 있음/없음

## 결정 사항
- (의도적으로 선택한 것, 트레이드오프)

## 다음 단계 제안
- (사용자가 다음에 할 만한 것)

## 주의 사항
- (사용자가 알아야 할 위험 또는 미완료 항목)
```

## 6. 막히면 할 일

- 자가 추측으로 진행하지 말 것.
- 인프라 정보는 `docs/infrastructure.md`에 다 있다. 거기 먼저 보기.
- 코드 규약은 `docs/INDEX.md` → 해당 문서.
- 그래도 없으면 사용자에게 명확한 질문으로 확인.
- 같은 실수가 반복되면 `docs/decisions/`에 ADR로 기록.

---

## 7. 자세한 규약

상세 규약은 다음 문서로 분리되어 있다. **필요할 때만** 읽는다:

- **인프라/접속 정보 → `docs/infrastructure.md`** ⭐
- 폴더 구조, 네이밍 → `docs/conventions.md`
- 아키텍처 결정 → `docs/architecture.md`
- 도메인 용어 → `docs/domain-glossary.md`
- 과거 결정 기록 → `docs/decisions/`

> 이 헌법은 살아있는 문서다. 규칙이 모호하거나 충돌하면 사용자에게 보고하고 업데이트를 제안할 것.
